Rappel & défi du jour

Ce qu'on a fait en S1

  • Construire une app Kanban locale

  • Obtenir du code fonctionnel… en tâtonnant

Le problème ressenti

  • Claude "part dans tous les sens"
  • On ne sait pas ce qu'il fait
  • Il ne fait pas toujours la même chose

Défi du jour : reconstruire une app Kanban depuis zéro — mais cette fois, en gardant le contrôle à chaque étape.

Le problème : Claude qui improvise

🎲

Quand on donne une tâche complexe à Claude sans le cadrer, il commence immédiatement à écrire du code.

  • Il fait des choix arbitraires de structure
  • Il oublie des fonctionnalités demandées
  • Une fois le code écrit, difficile de revenir en arrière

Il ne manque pas d'intelligence — il manque d'un moment pour réfléchir avant d'agir.

/plan — comment ça marche

Deux façons d'activer le mode planification :

Option A — dans la requête

Ajouter /plan à la fin de n'importe quelle requête

Option B — changer le mode

Appuyer sur Shift-Tab (Windows & Mac) pour basculer Claude Code en mode Plan avant d'écrire quoi que ce soit

  • Dans les deux cas, Claude réfléchit et propose une approche — sans écrire une seule ligne de code
  • On valide (ou modifie) le plan, puis on dit "Go" pour lancer l'exécution

Avantage : on reste décisionnaire. Claude ne fait rien sans notre accord explicite.

Analogie : c'est comme demander à un architecte de vous montrer les plans avant de commencer les travaux.

Exercice 1 10 min

Planifier l'app Kanban

Requête (en anglais)
I want to build a Kanban board app in HTML/CSS/JS (no framework).
It should have columns (To Do, In Progress, Done) and draggable
cards with a title and description. Keep it minimal — no priority
levels, labels, or extra features for now. Create all files in a
subdirectory called kanban/. /plan
  1. Ouvrez Claude Code dans votre terminal
  2. Copiez et envoyez la requête ci-dessus
  3. Lisez le plan — ne validez pas encore

Attendez avant de continuer !

Après avoir affiché le plan, Claude Code peut proposer de l'exécuter immédiatement — sous deux formes selon vos skills :

Claude has written up a plan and is ready to execute.
Would you like to proceed?
❯ 1. Yes, clear context (18% used) and auto-accept edits
  2. Yes, auto-accept edits
  3. Yes, manually approve edits
  4. Type here to tell Claude what to change

Ou, si le skill executing-plans est actif :

❯ 1. Subagent-Driven (this session) — fresh subagent per task, review between tasks
  2. Parallel Session (separate) — open new session with executing-plans skill

Ne choisissez aucune option pour l'instant. On activera Ctrl-O d'abord dans l'exercice suivant — sans ça, les premières actions partiront sans surveillance.

Débrief — /plan

Ce que vous avez observé :

  • Claude a listé les fichiers à créer, les fonctionnalités, l'ordre de travail
  • Il a attendu votre validation avant d'écrire quoi que ce soit
  • Modifier le plan avant l'exécution coûte 30 secondes — modifier le code après peut coûter 10 minutes

À retenir : /plan déplace la décision du code vers l'intention. Vous êtes le chef de projet, Claude est l'exécutant.

Ctrl-O : que fait Claude vraiment ?

🔍

Vous avez votre plan — parfait. Mais quand Claude va l'exécuter, il va effectuer des actions concrètes sur votre système, dans l'ordre qu'il choisit.

  • Il lit des fichiers existants
  • Il crée de nouveaux fichiers
  • Il modifie du code
  • Il peut exécuter des commandes terminal

Par défaut, tout ça se passe sans que vous voyiez quoi que ce soit. Vous découvrez le résultat, pas le processus.

Ctrl-O change ça : il rend chaque action visible et demande votre approbation avant de l'exécuter.

Ctrl-O — les actions de Claude

Appuyer sur Ctrl-O (Windows & Mac) active le mode "verbose" : chaque action est affichée avant d'être exécutée.

Les quatre grandes catégories d'actions :

Read Lire un fichier existant

Write Créer un nouveau fichier

Edit Modifier un fichier existant

Bash Exécuter une commande terminal

Vous pouvez approuver ou refuser chaque action individuellement. Ctrl-O rend Claude transparent.

Exercice 2 7 min

Exécuter le plan sous surveillance

  1. Appuyez sur Ctrl-O dans Claude Code — avant de faire quoi que ce soit d'autre
  2. Reprenez votre conversation de l'exercice 1 et lancez l'exécution — soit en tapant "Go", soit en choisissant une option proposée par Claude (ex. Yes, auto-accept edits ou Subagent-Driven)
  3. Pour chaque action affichée, identifiez son type : Read Write Edit Bash
  4. Approuvez les actions une par une — ou refusez-en une pour voir ce qui se passe

Important : activez Ctrl-O avant de valider le plan, sinon les premières actions partiront sans surveillance.

Débrief — Ctrl-O

Ce que vous avez observé :

  • Chaque outil utilisé par Claude est nommé et justifié
  • Vous pouvez dire NON à une action spécifique
  • Claude n'est pas une "boîte noire" — ses actions sont lisibles

Sans Ctrl-O

On approuve en aveugle. On découvre les problèmes trop tard.

Avec Ctrl-O

On voit, on comprend, on valide en connaissance de cause.

Le problème : des comportements à répéter

🔄

Certaines bonnes pratiques, on veut que Claude les applique systématiquement :

  • Toujours explorer les alternatives avant de choisir
  • Toujours écrire les tests avant le code
  • Toujours vérifier la cohérence avant de livrer

Si on doit le répéter à chaque requête, c'est du temps perdu — et on finit par oublier.

Les skills : l'analogie de la recette

Un skill, c'est une recette de cuisine.
On l'écrit une fois, on l'invoque à volonté. Claude suit les étapes à la lettre.

  • Un skill est un fichier texte (Markdown) rangé dans ~/.claude/skills/
  • Il décrit un comportement que Claude doit adopter
  • On l'invoque avec le mot-clé défini dans le fichier
  • Claude le lit et l'applique comme une instruction permanente pour la tâche

Des skills pré-installés existent déjà : simplify, systematic-debugging, tdd…

Démo formateur 3 min

Le skill "simplify" en live

Le formateur montre en direct :

  1. Ouvrir la conversation de l'exercice 2 — le code Kanban est déjà là
  2. Invoquer le skill simplify → Claude relit le code et identifie ce qui peut être simplifié, réutilisé ou amélioré
  3. Observer : Claude ne réécrit pas tout — il cible précisément, avec une justification pour chaque changement

Ce qu'on observe : sans le skill, Claude aurait pu réécrire le tout. Avec le skill, il applique une grille de lecture précise — la même à chaque fois.

/clear — pourquoi vider le contexte ?

À chaque échange, Claude "voit" l'intégralité de la conversation : vos messages, ses réponses, le code produit, les actions Ctrl-O… C'est le contexte.

Contexte trop long

  • Réponses plus lentes
  • Coût API plus élevé (plus de tokens traités)
  • Claude perd le fil, se contredit

Après /clear

  • Contexte remis à zéro
  • Réponses plus précises, plus rapides
  • Coût réduit dès le prochain message

/clear ne supprime rien sur le disque — vos fichiers sont intacts. Seule la mémoire de la conversation est effacée.

Bonne pratique : utilisez /clear entre deux tâches distinctes, pas en plein milieu d'une tâche en cours.

Exercice 3 5 min

Invoquer le skill simplify

  1. Tapez /clear — la conversation se vide, mais vos fichiers restent intacts sur le disque
  2. Envoyez la requête suivante pour invoquer le skill sur le code existant :
Requête
Review the kanban app in the current directory. /simplify
  1. Claude relit le code et applique le skill
  2. Identifiez : qu'est-ce qu'il a ciblé ? Pourquoi ?
  3. Acceptez ou refusez les modifications proposées

/clear ne supprime pas les fichiers — il remet le contexte à zéro. C'est utile pour repartir proprement entre deux tâches.

Débrief — Skills

Ce que vous avez observé :

  • /clear vide la conversation — mais les fichiers sur disque sont inchangés
  • Claude a appliqué une grille précise — pas sa propre interprétation de "améliorer"
  • Le résultat est prévisible et reproductible : invoquer simplify demain donnera la même approche
  • Vous gardez le contrôle : vous acceptez ou refusez chaque modification

À retenir : un skill encode un comportement que vous voulez systématique. Écrit une fois, appliqué à la demande — toujours de la même façon. Et /clear permet de changer de tâche proprement.
